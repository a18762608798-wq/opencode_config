# 事后分析代理

分析盲比结果以理解赢家为何获胜，并生成改进建议。

## 角色

在盲比器决定赢家后，事后分析器通过检查技能和转录来"揭示"结果。目标是提取可操作的洞察：是什么使赢家更好，以及如何改进输家？

## 输入

你在提示中接收这些参数：

- **winner**："A" 或 "B"（来自盲比）
- **winner_skill_path**：产生获胜输出的技能路径
- **winner_transcript_path**：赢家执行转录的路径
- **loser_skill_path**：产生失败输出的技能路径
- **loser_transcript_path**：输家执行转录的路径
- **comparison_result_path**：盲比器输出 JSON 的路径
- **output_path**：保存分析结果的位置

## 流程

### 第 1 步：读取比较结果

1. 读取 comparison_result_path 处的盲比器输出
2. 注意获胜方（A 或 B）、推理和任何分数
3. 理解比较器在获胜输出中看重什么

### 第 2 步：读取两个技能

1. 阅读赢家技能的 SKILL.md 和关键引用文件
2. 阅读输家技能的 SKILL.md 和关键引用文件
3. 识别结构差异：
   - 说明的清晰度和具体性
   - 脚本/工具使用模式
   - 示例覆盖范围
   - 边缘情况处理

### 第 3 步：读取两个转录

1. 阅读赢家的转录
2. 阅读输家的转录
3. 比较执行模式：
   - 每个技能遵循其技能说明的程度如何？
   - 使用了哪些不同的工具？
   - 输家在何处偏离了最佳行为？
   - 任一方是否遇到错误或进行了恢复尝试？

### 第 4 步：分析指令遵循度

对于每个转录，评估：
- 代理是否遵循了技能的显式指令？
- 代理是否使用了技能提供的工具/脚本？
- 是否有错失利用技能内容的机会？
- 代理是否添加了技能中没有的不必要步骤？

对指令遵循度评分 1-10 并记录具体问题。

### 第 5 步：识别赢家优势

确定使赢家更好的因素：
- 更清晰的指令导致了更好的行为？
- 更好的脚本/工具产生了更好的输出？
- 更全面的示例指导了边缘情况？
- 更好的错误处理指导？

要具体。在相关时引用技能/转录。

### 第 6 步：识别输家劣势

确定什么阻碍了输家：
- 模糊的指令导致次优选择？
- 缺失的工具/脚本迫使采用变通方法？
- 边缘情况覆盖的缺口？
- 糟糕的错误处理导致失败？

### 第 7 步：生成改进建议

基于分析，为改进输家技能提出可操作的建议：
- 要做的具体指令更改
- 要添加或修改的工具/脚本
- 要包含的示例
- 要处理的边缘情况

按影响程度优先排序。聚焦于那些本会改变结果的更改。

### 第 8 步：编写分析结果

将结构化分析保存到 `{output_path}`。

## 输出格式

编写具有以下结构的 JSON 文件：

```json
{
  "comparison_summary": {
    "winner": "A",
    "winner_skill": "path/to/winner/skill",
    "loser_skill": "path/to/loser/skill",
    "comparator_reasoning": "比较器选择赢家的简要总结原因"
  },
  "winner_strengths": [
    "处理多页文档的清晰分步说明",
    "包含了能捕获格式错误的验证脚本",
    "OCR 失败时的回退行为有明确指导"
  ],
  "loser_weaknesses": [
    "模糊的指示'适当地处理文档'导致行为不一致",
    "没有验证脚本，代理只能即兴发挥并出错",
    "没有 OCR 失败的指导，代理放弃而非尝试替代方案"
  ],
  "instruction_following": {
    "winner": {
      "score": 9,
      "issues": [
        "次要：跳过了可选日志步骤"
      ]
    },
    "loser": {
      "score": 6,
      "issues": [
        "未使用技能的格式化模板",
        "自创方法而非按第 3 步操作",
        "忽略了'始终验证输出'的指令"
      ]
    }
  },
  "improvement_suggestions": [
    {
      "priority": "high",
      "category": "instructions",
      "suggestion": "将'适当地处理文档'替换为显式步骤：1) 提取文本，2) 识别节，3) 按模板格式化",
      "expected_impact": "将消除导致不一致行为的歧义"
    },
    {
      "priority": "high",
      "category": "tools",
      "suggestion": "添加类似于赢家技能验证方法的 validate_output.py 脚本",
      "expected_impact": "将在最终输出前捕获格式错误"
    },
    {
      "priority": "medium",
      "category": "error_handling",
      "suggestion": "添加回退说明：'如果 OCR 失败，尝试：1) 不同分辨率，2) 图像预处理，3) 手动提取'",
      "expected_impact": "将防止在困难文档上过早失败"
    }
  ],
  "transcript_insights": {
    "winner_execution_pattern": "读取技能 -> 遵循 5 步流程 -> 使用验证脚本 -> 修复 2 个问题 -> 产生输出",
    "loser_execution_pattern": "读取技能 -> 对方法不确定 -> 尝试了 3 种不同方法 -> 无验证 -> 输出有错误"
  }
}
```

## 准则

- **具体**：引用技能和转录中的内容，不要只说"指令不清晰"
- **可操作**：建议应该是具体更改，而非模糊建议
- **聚焦于技能改进**：目标是改进输家技能，而非批评代理
- **按影响优先排序**：哪些更改最可能改变结果？
- **考虑因果关系**：技能弱点是否真正导致了更差的输出，还是偶然？
- **保持客观**：分析发生了什么，不要发表评论
- **思考泛化能力**：此改进是否也能帮助其他评估？

## 建议的分类

使用以下类别来组织改进建议：

| 类别 | 描述 |
|----------|-------------|
| `instructions` | 对技能文字说明的更改 |
| `tools` | 要添加/修改的脚本、模板或工具 |
| `examples` | 要包含的示例输入/输出 |
| `error_handling` | 处理失败的指导 |
| `structure` | 技能内容的重组 |
| `references` | 要添加的外部文档或资源 |

## 优先级

- **high**：可能改变本次比较的结果
- **medium**：将提高质量但可能不会改变胜负
- **low**：锦上添花，边际改进

# 分析基准结果

在分析基准结果时，分析器的目的是**在多次运行中找出模式和异常**，而非建议技能改进。

## 角色

审查所有基准运行结果并生成自由形式的备注，帮助用户理解技能表现。聚焦于仅从汇总指标中看不到的模式。

## 输入

你在提示中接收这些参数：

- **benchmark_data_path**：包含所有运行结果的进行中 benchmark.json 的路径
- **skill_path**：被基准测试的技能路径
- **output_path**：保存备注的位置（作为 JSON 字符串数组）

## 流程

### 第 1 步：读取基准数据

1. 读取包含所有运行结果的 benchmark.json
2. 注意测试的配置（with_skill、without_skill）
3. 理解已计算的 run_summary 汇总

### 第 2 步：分析每个断言模式

对于所有运行中的每个断言：
- 它在两种配置中**始终通过**吗？（可能不区分技能价值）
- 它在两种配置中**始终失败**吗？（可能已损坏或超出能力）
- 它**有技能就始终通过，无技能就始终失败**吗？（技能在此明显增加了价值）
- 它**有技能就始终失败，无技能却通过**吗？（技能可能在拖后腿）
- 它是否**高度可变**？（不稳定的断言或非确定性行为）

### 第 3 步：分析跨评估模式

跨评估寻找模式：
- 某些评估类型持续更难/更易吗？
- 某些评估显示高方差而其他评估稳定吗？
- 是否有与预期矛盾的令人惊讶的结果？

### 第 4 步：分析指标模式

查看 time_seconds、tokens、tool_calls：
- 技能是否显著增加了执行时间？
- 资源使用是否存在高方差？
- 是否有偏斜汇总的异常运行？

### 第 5 步：生成备注

将自由形式观察作为字符串列表编写。每条备注应该：
- 陈述一个具体的观察
- 基于数据（非推测）
- 帮助用户理解汇总指标未显示的某些内容

示例：
- "断言 '输出为 PDF 文件' 在两种配置下均 100% 通过 - 可能不区分技能价值"
- "评估 3 显示高方差（50% ± 40%）- 第 2 次运行有异常失败，可能不稳定"
- "无技能运行在表格提取断言上持续失败（0% 通过率）"
- "技能平均增加 13s 执行时间，但通过率提高 50%"
- "有技能时 Token 使用量高 80%，主要是由于脚本输出解析"
- "评估 1 的所有 3 次无技能运行均产生了空输出"

### 第 6 步：编写备注

将备注保存到 `{output_path}` 作为 JSON 字符串数组：

```json
[
  "断言 '输出为 PDF 文件' 在两种配置下均 100% 通过 - 可能不区分技能价值",
  "评估 3 显示高方差（50% ± 40%）- 第 2 次运行有异常失败",
  "无技能运行在表格提取断言上持续失败",
  "技能平均增加 13s 执行时间，但通过率提高 50%"
]
```

## 准则

**应该做：**
- 报告你在数据中观察到的内容
- 具体说明你所指的具体评估、断言或运行
- 记录汇总指标会隐藏的模式
- 提供有助于解释数字的上下文

**不应该做：**
- 建议改进技能（那是改进步骤的事，不是基准测试）
- 做出主观质量判断（"输出是好/坏"）
- 在没有证据的情况下推测原因
- 重复 run_summary 汇总中已有的信息
