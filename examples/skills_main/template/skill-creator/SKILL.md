# 技能创建器

一个用于创建新技能并迭代改进它们的技能。

概括来说，创建技能的过程如下：

- 确定你想让技能做什么以及大致如何做
- 编写技能草稿
- 创建几个测试提示，并用能访问该技能的 OpenCode 在其上运行
- 帮助用户定性和定量地评估结果
  - 在运行进行中的同时，如果没有的话起草一些定量评估（如果有现成的，可以直接使用，或如果觉得需要更改就修改）。然后向用户解释（如果已存在，则解释已有的评估）
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果以供查看，同时也让他们查看定量指标
- 根据用户对结果的评估反馈重写技能（以及如果从定量基准中发现明显缺陷的话）
- 重复直到满意
- 扩展测试集并在更大规模上重试

你在此技能中的工作是想出用户处于此过程中的哪个阶段，然后介入并帮助他们完成这些阶段。所以例如，可能他们像是"我想为 X 创建一个技能"。你可以帮助缩小他们想法的范围，编写草稿，编写测试用例，想出评估方式，运行所有提示，然后重复。

另一方面，也许他们已经有一个技能草稿。在这种情况下可以直接进入评估/迭代循环。

当然，你应该始终灵活应变，如果用户像是"我不需要运行大量评估，跟我配合着调整就行"，你也可以那样做。

然后技能完成后（但同样，顺序是灵活的），你也可以运行技能描述优化器，我们有一个单独的脚本来优化技能的触发。

明白了吗？好。

## 与用户沟通

关注上下文线索来理解决如何措辞你的沟通！在默认情况下，给你一些大致概念：

- "评估"和"基准"处在边界线上，但可以接受
- 对于"JSON"和"断言"，在没有解释的情况下使用它们之前，你希望看到用户了解这些术语的严肃线索

如果有疑问，可以简短地解释术语，如果不确定用户是否能理解，随时可以用简短的定义进行澄清。

## 创建技能

### 捕捉意图

从理解用户意图开始。当前对话可能已经包含用户想要捕捉的工作流（例如，他们说"把这个变成技能"）。如果是这样，先从对话历史中提取答案——使用的工具、步骤序列、用户做出的修正、观察到的输入/输出格式。用户可能需要填补空白，在进入下一步之前应确认。

1. 此技能应使 OpenCode 能够做什么？
2. 此技能何时触发？（什么用户短语/上下文）
3. 预期的输出格式是什么？
4. 是否应该设置测试用例来验证技能是否有效？具有客观可验证输出的技能（文件转换、数据提取、代码生成、固定工作流步骤）受益于测试用例。具有主观输出的技能（写作风格、艺术）通常不需要。根据技能类型建议适当的默认值，但让用户决定。

### 访谈与研究

主动询问关于边缘情况、输入/输出格式、示例文件、成功标准和依赖的问题。在弄清这些之前，不要急于编写测试提示。

检查可用的 MCP——如果对研究有用（搜索文档、查找类似技能、查阅最佳实践），在可用时通过子代理并行研究，否则内联进行。带着准备好的上下文来减轻用户的负担。

### 编写 SKILL.md

根据用户访谈，填写以下组件：

- **name**：技能标识符
- **description**：何时触发、做什么。这是主要的触发机制——包含技能做什么和何时使用它的具体上下文。所有"何时使用"的信息都在这里，不在正文中。注意：当前 OpenCode 倾向于"欠触发"技能——在技能有用时不使用它们。为应对此问题，请让技能描述稍微"积极"一些。例如，与其写"如何构建一个简单快速的仪表盘来显示内部 Anthropic 数据"，你可以写"如何构建一个简单快速的仪表盘来显示内部 Anthropic 数据。确保在用户提到仪表盘、数据可视化、内部指标或想要显示任何类型的公司数据时使用此技能，即使他们没有明确要求一个'仪表盘'。"
- **compatibility**：必需的工具、依赖（可选，很少需要）
- **技能的其余部分 :)**

### 技能编写指南

#### 技能的组成部分

```
skill-name/
├── SKILL.md（必需）
│   ├── YAML frontmatter（name、description 必需）
│   └── Markdown 指令
└── 打包资源（可选）
    ├── scripts/    - 用于确定性/重复任务的可执行代码
    ├── references/ - 按需加载到上下文的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式内容展开

技能使用三级加载系统：
1. **元数据**（名称 + 描述） - 始终在上下文中（约 100 词）
2. **SKILL.md 正文** - 技能触发时在上下文中（理想 < 500 行）
3. **打包资源** - 按需使用（大小不受限制，脚本可以执行而无需加载）

这些字数是大致参考，如果需要可以更长。

**关键模式：**
- 保持 SKILL.md 在 500 行以下；如果接近此限制，增加额外的层级并附上清晰指示，说明使用该技能的模型应前往何处继续。
- 从 SKILL.md 中清晰地引用文件，并指导何时阅读它们
- 对于大型引用文件（>300 行），包含目录

**领域组织**：当技能支持多个领域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md（工作流 + 选择）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
OpenCode 仅读取相关的引用文件。

#### 不放意外原则

这应该不言自明，但技能不得包含恶意软件、漏洞利用代码或任何可能危及系统安全的内容。如果描述其意图，技能的内容不应让用户意外。不要配合创建误导性技能或旨在促进未授权访问、数据窃取或其他恶意活动的技能。不过"角色扮演成 XYZ"之类的是可以的。

#### 编写模式

在指令中优先使用祈使句形式。

**定义输出格式**——可以这样做：
```markdown
## 报告结构
始终使用此确切模板：
# [标题]
## 执行摘要
## 关键发现
## 建议
```

**示例模式**——包含示例是有用的。可以像这样格式化（但如果示例中有"Input"和"Output"，你可能想稍微变化一下）：
```markdown
## 提交消息格式
**示例 1：**
Input: 添加了带 JWT 令牌的用户认证
Output: feat(auth): 实现基于 JWT 的认证
```

### 写作风格

尝试向模型解释为什么事情很重要，而不是使用沉重的必须 MUST。运用心理理论，尝试让技能通用而非超狭隘地针对特定示例。先写草稿，然后以新的眼光审视并改进。

### 测试用例

编写技能草稿后，想出 2-3 个真实的测试提示——真实用户真正会说出的那种。与用户分享：[你不必使用此确切措辞]"这是我想要尝试的几个测试用例。这些看起来正确吗，还是你想添加更多？"然后运行它们。

将测试用例保存到 `evals/evals.json`。先不要写断言——只放提示。你将在下一步草拟断言，同时运行正在进行中。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示",
      "expected_output": "预期结果的描述",
      "files": []
    }
  ]
}
```

参见 `references/schemas.md` 获取完整 schema（包括 `assertions` 字段，稍后添加）。

## 运行和评估测试用例

本节是一个连续的序列——不要中途停止。不要使用 `/skill-test` 或任何其他测试技能。

将结果放在 `<skill-name>-workspace/` 中作为技能目录的同级。在工作区内，按迭代组织结果（`iteration-1/`、`iteration-2/` 等），在其中每个测试用例获得一个目录（`eval-0/`、`eval-1/` 等）。不要一开始就创建所有这些——只需在操作中按需创建目录。

### 第 1 步：在同一轮中启动所有运行（带技能和基线）

对于每个测试用例，在同一轮中启动两个子代理——一个有技能，一个没有。这很重要：不要先启动带技能运行，然后再回来做基线。一次性启动所有内容，让一切差不多同时完成。

**带技能运行：**

```
执行此任务：
- 技能路径：<path-to-skill>
- 任务：<eval prompt>
- 输入文件：<eval files 如果有，或 "none">
- 将输出保存到：<workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 要保存的输出：<用户关心的内容——例如 "the .docx file"、"the final CSV">
```

**基线运行**（相同提示，但基线取决于上下文）：
- **创建新技能**：完全没有技能。相同提示，无技能路径，保存到 `without_skill/outputs/`。
- **改进现有技能**：旧版本。在编辑之前，快照技能（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后将基线子代理指向快照。保存到 `old_skill/outputs/`。

为每个测试用例编写 `eval_metadata.json`（断言现在可以为空）。给每个评估一个描述性名称，基于它所测试的内容——不仅仅是"eval-0"。也使用此名称作为目录。如果此迭代使用新的或修改后的评估提示，为每个新评估目录创建这些文件——不要假设它们从先前的迭代中继承。

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "用户的任务提示",
  "assertions": []
}
```

### 第 2 步：在运行进行中时起草断言

不要只等待运行完成——你可以有效利用这段时间。为每个测试用例起草定量断言并向用户解释。如果 `evals/evals.json` 中已有断言，审查它们并解释它们检查什么。

好的断言是客观可验证的，并有描述性的名称——它们应在基准查看器中清晰易读，让人一眼就能理解每个断言检查什么。主观技能（写作风格、设计质量）更适合定性评估——不要对需要人类判断的东西强加断言。

一旦起草完成，用断言更新 `eval_metadata.json` 文件和 `evals/evals.json`。也向用户解释他们将在查看器中看到什么——定性输出和定量基准。

### 第 3 步：当运行完成时捕获时间数据

当每个子代理任务完成时，你会收到一个包含 `total_tokens` 和 `duration_ms` 的通知。立即将此数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会——它通过任务通知传递，不会在其他地方持久化。随着每个通知的到来处理它，而不是试图批量处理。

### 第 4 步：评分、聚合并启动查看器

一旦所有运行完成：

1. **对每次运行评分**——启动一个评分子代理（或内联评分），读取 `agents/grader.md` 并根据输出评估每个断言。将结果保存到每次运行目录中的 `grading.json`。grading.json 的 expectations 数组必须使用字段 `text`、`passed` 和 `evidence`（而非 `name`/`met`/`details` 或其他变体）——查看器依赖于这些确切字段名。对于可编程检查的断言，编写并运行脚本而非肉眼判断——脚本更快、更可靠且可跨迭代重用。

2. **聚合成基准**——从 skill-creator 目录运行聚合脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   这会产生 `benchmark.json` 和 `benchmark.md`，包含每个配置的 pass_rate、时间和 token 以及均值 ± 标准差和差值。如果手动生成 benchmark.json，参见 `references/schemas.md` 获取查看器期望的确切 schema。
   将每个带技能版本放在其基线对应版本之前。

3. **进行分析**——读取基准数据并发现聚合统计可能隐藏的模式。参见 `agents/analyzer.md`（"分析基准结果"部分）了解要找什么——诸如无论技能如何总是通过的断言（非区分性）、高方差评估（可能不稳定）以及时间/token 权衡。

4. **启动查看器**，包含定性输出和定量数据：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   对于迭代 2+，也传入 `--previous-workspace <workspace>/iteration-<N-1>`。

   **Cowork / 无头环境：** 如果 `webbrowser.open()` 不可用或环境没有显示，使用 `--static <output_path>` 写入独立 HTML 文件而非启动服务器。当用户点击"提交所有审查"时，反馈将下载为 `feedback.json` 文件。下载后，将 `feedback.json` 复制到工作区目录以供下一次迭代使用。

注意：请使用 generate_review.py 创建查看器；无需编写自定义 HTML。

5. **告诉用户**类似："我在你的浏览器中打开了结果。有两个标签页——'输出'让你点击浏览每个测试用例并留下反馈，'基准'显示定量对比。完成后，回到这里告诉我。"

### 用户在查看器中看到什么

"输出"标签一次显示一个测试用例：
- **提示**：给定的任务
- **输出**：技能生成的文件，在可能时内联渲染
- **上一次输出**（迭代 2+）：折叠部分，显示上一次迭代的输出
- **正式评分**（如果运行了评分）：折叠部分，显示断言通过/失败
- **反馈**：一个随输入自动保存的文本框
- **上一次反馈**（迭代 2+）：他们上一次的评论，显示在文本框下方

"基准"标签显示统计摘要：每个配置的通过率、时间和 token 使用情况，以及每项评估的细分和分析观察。

导航通过上一个/下一个按钮或箭头键。完成后，他们点击"提交所有审查"，将所有反馈保存到 `feedback.json`。

### 第 5 步：读取反馈

当用户告诉你他们完成了，读取 `feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "图表缺少轴标签", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完美，喜欢这个", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为没问题。将改进集中在用户有具体投诉的测试用例上。

完成后关闭查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你已运行测试用例，用户已审查结果，现在你需要根据他们的反馈让技能变得更好。

### 如何看待改进

1. **从反馈中泛化。** 这里要发生的大事是我们在尝试创建可以被使用一百万次的技能（可能字面意义上，甚至更多，谁知道呢）跨许多不同提示。在这里你和用户仅反复迭代几个示例，因为这有助于更快地推进。用户熟知这些示例，他们能快速评估新输出。但如果你和用户共同开发的技能只适用于这些示例，它就是无用的。与其放入琐碎的过拟性更改或压迫性的限制性 MUST，如果存在某些顽固问题，你可以尝试扩展并使用不同的比喻，或推荐不同的工作模式。尝试相对便宜，也许你会找到很好的方案。

2. **保持提示精简。** 移除那些没有实际价值的内容。确保阅读转录，而不仅仅是最终输出——如果看起来技能让模型浪费大量时间做无效的事情，你可以尝试去掉技能的那些部分看看会发生什么。

3. **解释为什么。** 努力解释你要求模型做的每件事的**为什么**。今天的 LLM 是*聪明的*。它们有良好的心理理论，当给予良好的框架时，可以超越机械指令，真正让事情发生。即使来自用户的反馈简洁或沮丧，尝试真正理解任务以及用户写他们所写内容的意图，他们真正写的是什么，然后将这种理解传递到指令中。如果你发现自己用全大写写 ALWAYS 或 NEVER，或使用超刚性的结构，那是黄旗——如果可能，重新构思并解释推理，让模型理解为什么你要求的内容很重要。这是更人性化、更强大、更有效的方法。

4. **在测试用例间寻找重复工作。** 阅读测试运行的转录，注意子代理是否都独立编写了相似的辅助脚本或对某事采取了相同的多步骤方法。如果所有 3 个测试用例都导致子代理编写 `create_docx.py` 或 `build_chart.py`，这是将其打包到技能中的强烈信号。写一次，放在 `scripts/` 中，并告诉技能使用它。这节省了每次调用的重复造轮子。

这项任务相当重要（我们正试图每年创造数十亿的经济价值！），你的思考时间不是瓶颈；慢慢来，真正地深入思考。我建议先写一个草稿修订，然后以全新的眼光审视并进行改进。真正尽力进入用户的头脑，理解他们想要和需要什么。

### 迭代循环

改进技能后：

1. 应用你的改进到技能
2. 将所有测试用例重新运行到一个新的 `iteration-<N+1>/` 目录中，包括基线运行。如果你正在创建新技能，基线始终是 `without_skill`（无技能）——这在各迭代间保持不变。如果你正在改进现有技能，自行判断什么作为基线有意义：用户带着的最初版本，或上一个迭代。
3. 使用 `--previous-workspace` 指向上一个迭代启动审查器
4. 等待用户审查并告诉你他们完成
5. 读取新反馈，再次改进，重复

持续进行，直到：
- 用户说他们满意
- 反馈全为空（一切看起来不错）
- 你没有取得有意义的进展

## 高级：盲比

对于需要更严格比较两个技能版本的情况（例如，用户问"新版本是否真的更好？"），有一个盲比系统。阅读 `agents/comparator.md` 和 `agents/analyzer.md` 了解详情。基本思想是：将两个输出传给一个独立的代理，不告诉它哪个是哪个，让它判断质量。然后分析为什么赢家赢了。

这是可选的，需要子代理，大多数用户不需要它。人工审查循环通常足够。

## 描述优化

SKILL.md frontmatter 中的 description 字段是决定 OpenCode 是否调用某个技能的主要机制。创建或改进技能后，提供优化描述以获得更好的触发准确性。

### 第 1 步：生成触发评估查询

创建 20 条评估查询——混合应触发和不应触发。保存为 JSON：

```json
[
  {"query": "用户提示", "should_trigger": true},
  {"query": "另一个提示", "should_trigger": false}
]
```

查询必须是真实的，是 OpenCode 用户真正会输入的内容。不是抽象请求，而是具体、具体、有相当多细节的请求。例如，文件路径、关于用户工作或个人情况的个人背景、列名和值、公司名称、URL。一点背景故事。有些可能是小写或包含缩写、拼写错误或随意用语。使用不同长度的混合，并侧重于边缘情况而非让它们泾渭分明（用户将有机会签署批准它们）。

坏例子：`"格式化此数据"`、`"从 PDF 提取文本"`、`"创建图表"`

好例子：`"ok 我老板刚刚把这个 xlsx 文件发给我了（在我的下载文件夹里，叫 'Q4 sales final FINAL v2.xlsx'），她想让我加一列显示利润率百分比。收入在 C 列，成本我觉得在 D 列"`

对于**应触发**查询（8-10 条），考虑覆盖面。需要相同意图的不同表述——有些正式，有些随意。包含用户没有明确命名技能或文件类型但明显需要它的情况。加入一些不常见的用例以及此技能与另一个竞争但应胜出的情况。

对于**不应触发**查询（8-10 条），最有价值的是近失误——与技能共享关键词或概念但实际上需要不同东西的查询。考虑相邻领域、朴素关键字匹配会触发但不应触发的模糊措辞，以及查询触及技能所做的某些事但在另一个工具更合适的上下文中。

关键要避免的：不要让不应触发查询明显不相关。`"写一个斐波那契函数"` 作为 PDF 技能的负面测试太容易了——它没有测试任何东西。负面用例应该真正有难度。

### 第 2 步：与用户审查

使用 HTML 模板向用户展示评估集以供审查：

1. 从 `assets/eval_review.html` 读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评估项的 JSON 数组（不加引号——这是一个 JS 变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → 技能名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能的当前描述
3. 写入临时文件（例如 `/tmp/eval_review_<skill-name>.html`）并打开：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换应触发状态、添加/删除条目，然后点击"导出评估集"
5. 文件下载到 `~/Downloads/eval_set.json`——检查 Downloads 文件夹以获取最新版本，以防有多个版本（例如 `eval_set (1).json`）

这一步很重要——糟糕的评估查询导致糟糕的描述。

### 第 3 步：运行优化循环

告诉用户："这需要一些时间——我将在后台运行优化循环并定期检查。"

将评估集保存到工作区，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

使用系统提示中的模型 ID（当前会话使用的那个），以便触发测试与用户实际体验的一致。

在运行时，定期 tail 输出告诉用户当前在哪个迭代以及得分情况。

这自动处理完整的优化循环。它将评估集分成 60% 训练和 40% 留出测试，评估当前描述（每个查询运行 3 次以获得可靠的触发率），然后调用 OpenCode 提出基于失败内容的改进建议。它重新评估每个新描述在训练集和测试集上的表现，最多迭代 5 次。完成后，在浏览器中打开显示每次迭代结果的 HTML 报告，并返回 JSON，其中 `best_description`——基于测试得分而非训练得分选择，以避免过拟合。

### 技能触发的工作原理

理解触发机制有助于设计更好的评估查询。技能出现在 OpenCode 的 `available_skills` 列表中，带有名称 + 描述，OpenCode 根据该描述决定是否查阅技能。重要的是，OpenCode 只为其自己不易处理的任务查阅技能——简单的单步查询如"读取此 PDF"可能不会触及相关技能，即使描述匹配完美，因为 OpenCode 可以用基础工具直接处理。复杂、多步骤或专门的查询在描述匹配时可靠地触发技能。

这意味着你的评估查询应该足够实质，让 OpenCode 确实能从查阅技能中受益。简单的查询如"读取文件 X"是糟糕的测试用例——无论描述质量如何，它们都不会触发技能。

### 第 4 步：应用结果

从 JSON 输出中获取 `best_description` 并更新技能的 SKILL.md frontmatter。向用户展示修改前后对比并报告得分。

### 打包和呈现（仅当 `present_files` 工具可用时）

检查你是否能访问 `present_files` 工具。如果不能，跳过此步骤。如果能，打包技能并将 .skill 文件呈现给用户：

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

打包后，指引用户到生成的 `.skill` 文件路径，以便他们安装。

## OpenCode Web 特定说明

在 OpenCode Web 中，核心工作流相同（草稿 → 测试 → 审查 → 改进 → 重复），但由于 OpenCode Web 没有子代理，某些机制有所变化。需要调整的内容：

**运行测试用例**：没有子代理意味着没有并行执行。对于每个测试用例，读取技能的 SKILL.md，然后按其指示自己完成测试提示。一次做一个。这没有独立子代理那么严格（你编写了技能并且你也在运行它，所以你有完整的上下文），但这是一个有用的健全性检查——而人工审查步骤进行了弥补。跳过基线运行——只需使用技能按要求完成任务。

**审查结果**：如果你打不开浏览器（例如 OpenCode Web 的 VM 没有显示，或你在远程服务器上），完全跳过浏览器查看器。相反，直接在对话中呈现结果。对于每个测试用例，展示提示和输出。如果输出是用户需要看到的文件（如 .docx 或 .xlsx），将其保存到文件系统并告诉他们位置，以便他们可以下载和检查。内联询问反馈："这个看起来怎么样？有需要更改的吗？"

**基准测试**：跳过定量基准测试——它依赖于基线对比，在没有子代理的情况下没有意义。关注于用户的定性反馈。

**迭代循环**：与之前相同——改进技能，重新运行测试用例，要求反馈——只是中间没有浏览器查看器。如果你有文件系统，仍然可以按迭代目录组织结果。

**描述优化**：此部分需要 `opencode` CLI 工具（具体是 `opencode -p`），它仅在 OpenCode 中可用。如果你在 OpenCode Web 上，跳过它。

**盲比**：需要子代理。跳过它。

**打包**：`package_skill.py` 脚本在 Python 和文件系统的任何地方都能运行。在 OpenCode Web 上，你可以运行它，用户可以下载生成的 `.skill` 文件。

**更新现有技能**：用户可能是要求你更新现有技能，而非创建新技能。在这种情况下：
- **保留原始名称。** 注意技能的目录名称和 `name` frontmatter 字段——原样使用它们。例如，如果已安装的技能是 `research-helper`，输出 `research-helper.skill`（而非 `research-helper-v2`）。
- **编辑前复制到可写位置。** 已安装的技能路径可能是只读的。复制到 `/tmp/skill-name/`，在那里编辑，并从副本打包。
- **如果手动打包，首先暂存在 `/tmp/` 中**，然后复制到输出目录——直接写入可能因权限而失败。

## Cowork 特定说明

如果你在 Cowork 中，主要须知：

- 你有子代理，因此主工作流（并行启动测试用例、运行基线、评分等）都可用。（然而，如果你在超时上遇到严重问题，可以按顺序而非并行运行测试提示。）
- 你没有浏览器或显示，因此在生成评估查看器时，使用 `--static <output_path>` 写入独立 HTML 文件而非启动服务器。然后提供一个链接，用户可以点击在浏览器中打开该 HTML。
- 出于某种原因，Cowork 设置似乎让 OpenCode 在运行测试后不愿生成评估查看器，因此再次强调：无论你是在 Cowork 还是 OpenCode 中，运行测试后，你应始终生成评估查看器供人工在修订技能之前查看示例，使用 `generate_review.py`（而非编写你自己的自定义 html 代码）。抱歉提前说，但我要全大写了：在自己评估输入之前先生成评估查看器。你想要尽快让他们看到！
- 反馈工作方式不同：由于没有运行的服务器，查看器的"提交所有审查"按钮将下载 `feedback.json` 作为文件。然后你可以从那里读取（你可能需要先请求访问权限）。
- 打包可以工作——`package_skill.py` 仅需要 Python 和文件系统。
- 描述优化（`run_loop.py` / `run_eval.py`）应在 Cowork 中正常工作，因为它通过子进程使用 `claude -p` 而非浏览器，但请保存它直到你完全完成技能并且用户同意它处于良好状态。
- **更新现有技能**：用户可能是要求你更新现有技能，而非创建新技能。遵循上述 claude.ai 部分中的更新指南。
