# 文档协作编写工作流

此技能提供一个结构化的工作流，引导用户进行协作式文档创建。作为主动引导者，引导用户完成三个阶段：上下文收集、精炼与结构化，以及读者测试。

## 何时提供此工作流

**触发条件：**
- 用户提到编写文档："写一个文档"、"起草提案"、"创建规格"、"撰写"
- 用户提到特定文档类型："PRD"、"设计文档"、"决策文档"、"RFC"
- 用户似乎正在开始一项实质性的写作任务

**初始提议：**
向用户提供一个结构化的文档协作编写工作流。解释三个阶段：

1. **上下文收集**：用户提供所有相关上下文，OpenCode 同时提出澄清问题
2. **精炼与结构化**：通过头脑风暴和编辑逐节构建文档
3. **读者测试**：用一个新的 OpenCode（无上下文）测试文档，在他人阅读之前发现盲点

解释这种方法有助于确保文档在他人阅读时有效（包括他们将其粘贴到 OpenCode 中时）。询问他们是想尝试此工作流还是自由形式进行。

如果用户拒绝，自由形式进行。如果接受，进入阶段 1。

## 阶段 1：上下文收集

**目标：** 缩小用户所知与 OpenCode 所知之间的差距，以便后续提供智能指导。

### 初始问题

首先询问用户关于文档的元上下文：

1. 这是什么类型的文档？（例如，技术规格、决策文档、提案）
2. 主要受众是谁？
3. 当有人阅读此文档时，期望达到什么影响？
4. 是否有模板或特定格式需要遵循？
5. 还有其他约束或上下文需要了解吗？

告知他们可以用简写回答，或以任何对他们最有效的方式倾卸信息。

**如果用户提供模板或提到文档类型：**
- 询问他们是否有模板文档可以分享
- 如果提供共享文档的链接，使用适当的集成获取它
- 如果提供了文件，读取它

**如果用户提到编辑现有的共享文档：**
- 使用适当的集成读取当前状态
- 检查没有替代文本的图片
- 如果存在没有替代文本的图片，说明当他人通过 OpenCode 理解文档时，OpenCode 将无法看到它们。询问是否需要生成替代文本。如果需要，请求他们将每张图片粘贴到对话中以生成描述性替代文本。

### 信息倾卸

一旦初始问题得到回答，鼓励用户倾卸他们所拥有的所有上下文。请求提供如下信息：
- 项目/问题的背景
- 相关的团队讨论或共享文档
- 为什么没有使用替代方案
- 组织上下文（团队动态、过往事故、政治因素）
- 时间线压力或约束
- 技术架构或依赖
- 利益相关方关注点

建议他们不用担心组织——只需要全部吐出来。提供多种提供上下文的方式：
- 意识流式信息倾卸
- 指向团队频道或讨论串进行读取
- 链接到共享文档

**如果有集成可用**（例如 Slack、Teams、Google Drive、SharePoint 或其他 MCP 服务器），提到可以利用这些来直接拉取上下文。

**如果没有检测到集成且在 OpenCode Web 或 OpenCode 应用中：** 建议他们可以在 OpenCode 设置中启用连接器，以允许直接从消息应用和文档存储中拉取上下文。

告知他们在完成初始倾卸后将会提出澄清问题。

**在上下文收集期间：**

- 如果用户提到团队频道或共享文档：
  - 如果有集成可用：告知内容即将被读取，然后使用适当的集成
  - 如果没有集成可用：说明缺乏访问权限。建议他们在 OpenCode 设置中启用连接器，或直接粘贴相关内容。

- 如果用户提到未知的实体/项目：
  - 询问是否应该搜索连接的工具以了解更多
  - 在搜索之前等待用户确认

- 当用户提供上下文时，跟踪学到什么和仍然不清楚的内容

**提出澄清问题：**

当用户表明已完成初始倾卸（或在提供了大量上下文之后），提出澄清问题以确保理解：

根据上下文中的空白生成 5-10 个带编号的问题。

告知他们可以用简写回答（例如，"1：是，2：见 #channel，3：否，因为向后兼容"），链接到更多文档，指向需要读取的频道，或继续信息倾卸。哪种方式对他们最有效就用哪种。

**退出条件：**
当提问表明你已经理解——可以询问边缘情况和权衡而不需要解释基础内容时，表明已收集足够的上下文。

**过渡：**
询问他们是否在此阶段还有更多上下文要提供，还是该进入文档起草阶段。

如果用户想添加更多，让他们添加。准备好后，进入阶段 2。

## 阶段 2：精炼与结构化

**目标：** 通过头脑风暴、策划和迭代精炼，逐节构建文档。

**对用户的说明：**
解释将逐节构建文档。对于每节：
1. 将询问关于应包含什么的澄清问题
2. 将头脑风暴 5-20 个选项
3. 用户将指示保留/删除/合并什么
4. 将起草该节
5. 通过精准编辑进行精炼

从未知最多的一节开始（通常是核心决策/提案），然后处理其余部分。

**节顺序：**

如果文档结构清晰：
询问他们想从哪一节开始。

建议从未知最多的一节开始。对于决策文档，通常是核心提案。对于规格，通常是技术方案。摘要部分最好留到最后。

如果用户不知道需要哪些节：
根据文档类型和模板，建议 3-5 个适合该文档类型的节。

询问这个结构是否可行，或他们是否想调整。

**一旦结构达成一致：**

创建初始文档结构，所有节都有占位文本。

**如果可以访问 artifacts：**
使用 `create_file` 创建 artifact。这给了 OpenCode 和用户一个可用的脚手架。

告知他们将创建包含所有节标题和简短占位文本的初始结构，如"[待编写]"或"[内容在此]"。

使用所有节标题和简要占位文本创建 artifact。

提供脚手架链接，并表明该填充每个节了。

**如果无法访问 artifacts：**
在工作目录中创建一个 markdown 文件。根据文档类型适当命名（例如 `decision-doc.md`、`technical-spec.md`）。

告知他们将创建包含所有节标题和占位文本的初始结构。

创建包含所有节标题和占位文本的文件。

确认文件名已创建，并表明该填充每个节了。

**对于每个节：**

### 第 1 步：澄清问题

宣布将开始处理 [节名称] 节。提出 5-10 个关于应包含什么的澄清问题：

根据上下文和节的目的生成 5-10 个具体问题。

告知他们可以用简写回答，或仅指出哪些内容重要需要涵盖。

### 第 2 步：头脑风暴

对于 [节名称] 节，头脑风暴 [5-20] 个可能包含的内容，取决于节的复杂度。寻找：
- 可能已被遗忘的共享上下文
- 尚未提到的角度或考虑

根据节的复杂度生成 5-20 个编号选项。最后，如果他们需要更多选项，提出可以进一步头脑风暴。

### 第 3 步：策划

询问哪些要点应该保留、删除或合并。请求简要的理由，以帮助了解后续各节中的优先级。

提供示例：
- "保留 1、4、7、9"
- "删除 3（与 1 重复）"
- "删除 6（受众已知）"
- "合并 11 和 12"

**如果用户给出自由形式反馈**（例如，"看起来不错"或"我大部分喜欢但是……"）而非编号选择，提取他们的偏好并继续。解析他们想要保留/删除/更改的内容并应用。

### 第 4 步：缺口检查

根据他们选择的内容，询问 [节名称] 节是否有重要的遗漏。

### 第 5 步：起草

使用 `str_replace` 将此节的占位文本替换为实际起草的内容。

宣布 [节名称] 节将根据他们选择的内容起草。

**如果使用 artifacts：**
起草后，提供 artifact 的链接。

请他们通读并指出需要更改的地方。注意具体反馈有助于为后续各节学习。

**如果使用文件（无 artifacts）：**
起草后，确认完成。

告知他们 [节名称] 节已在 [文件名] 中起草。请他们通读并指出需要更改的地方。注意具体反馈有助于为后续各节学习。

**起草第一节时给用户的关键说明：**
提供说明：与其直接编辑文档，请他们指出需要更改的地方。这有助于学习他们的风格以便后续各节。例如："删除 X 项——已被 Y 涵盖"或"让第三段更简洁"。

### 第 6 步：迭代精炼

当用户提供反馈时：
- 使用 `str_replace` 进行编辑（绝不重印整个文档）
- **如果使用 artifacts：** 每次编辑后提供 artifact 链接
- **如果使用文件：** 仅确认编辑完成
- 如果用户直接编辑文档并要求阅读：在心中记下他们所做的更改，并在后续各节中牢记（这展示了他们的偏好）

**持续迭代**直到用户对该节满意。

### 质量检查

在连续 3 次迭代没有实质性更改后，询问是否有内容可以删除而不丢失重要信息。

当本节完成时，确认 [节名称] 已完成。询问是否准备好进入下一节。

**对所有节重复。**

### 接近完成

当接近完成（80% 以上的节已完成）时，宣布打算重读整个文档并检查：
- 各节间的流畅性和一致性
- 冗余或矛盾
- 任何感觉像是"套路"或通用填充的内容
- 每个句子是否都有分量

阅读整个文档并提供反馈。

**当所有节都起草并精炼完毕：**
宣布所有节已起草。表示打算再次审查完整文档。

审查整体连贯性、流畅性、完整性。

提供任何最终建议。

询问是否准备进入读者测试，或还想精炼其他内容。

## 阶段 3：读者测试

**目标：** 用一个全新的 OpenCode（无上下文渗透）测试文档，以验证其对读者有效。

**对用户的说明：**
解释现在将测试文档是否对读者真正有效。这能发现盲点——那些对作者有意义的但对他人来说可能困惑的内容。

### 测试方法

**如果可以访问子代理（例如，在 OpenCode 中）：**

直接执行测试，无需用户参与。

### 第 1 步：预测读者问题

宣布打算预测读者在尝试发现此文档时可能提出的问题。

生成 5-10 个读者会真实提出的问题。

### 第 2 步：使用子代理测试

宣布将使用一个全新的 OpenCode 实例（无此对话上下文）来测试这些问题。

对于每个问题，调用一个子代理，仅传入文档内容和问题。

总结读者 OpenCode 对每个问题的正确/错误回答。

### 第 3 步：执行额外检查

宣布将执行额外检查。

调用子代理检查模糊性、错误假设和矛盾。

总结发现的任何问题。

### 第 4 步：报告和修复

如果发现问题：
报告读者 OpenCode 在特定问题上遇到困难。

列出具体问题。

表示打算修复这些缺口。

循环回到问题节的精炼阶段。

---

**如果无法访问子代理（例如，claude.ai Web 界面）：**

用户需要手动进行测试。

### 第 1 步：预测读者问题

询问人们在尝试发现此文档时可能提出什么问题。他们会在 OpenCode 中输入什么？

生成 5-10 个读者会真实提出的问题。

### 第 2 步：设置测试

提供测试说明：
1. 打开一个新的 OpenCode 对话：https://claude.ai
2. 粘贴或分享文档内容（如果使用启用了连接器的共享文档平台，提供链接）
3. 向读者 OpenCode 提问生成的问题

对于每个问题，指示读者 OpenCode 提供：
- 答案
- 任何模糊或不清晰的内容
- 文档假定了哪些知识/上下文

检查读者 OpenCode 是否给出正确答案或误解了任何内容。

### 第 3 步：额外检查

也向读者 OpenCode 提问：
- "此文档中哪些内容可能对读者来说是模糊或不清晰的？"
- "此文档假定了读者已经具备哪些知识或上下文？"
- "是否存在任何内部矛盾或不一致？"

### 第 4 步：根据结果迭代

询问读者 OpenCode 哪些地方理解错误或遇到困难。表示打算修复这些缺口。

循环回到问题节的精炼阶段。

---

### 退出条件（两种方法）

当读者 OpenCode 持续正确回答问题且没有出现新的缺口或模糊时，文档就准备好了。

## 最终审查

当读者测试通过时：
宣布文档已通过读者 OpenCode 测试。完成之前：

1. 建议他们自己再通读一次——这份文档由他们拥有，对其质量负责
2. 建议再次检查所有事实、链接或技术细节
3. 请他们验证文档是否达到了他们期望的影响

询问是否需要再进行一次审查，还是工作已完成。

**如果用户想要最终审查，提供它。否则：**
宣布文档完成。提供一些最终建议：
- 考虑在附录中链接此对话，以便读者了解文档是如何开发的
- 使用附录提供深度而不让主体文件臃肿
- 当收到真实读者反馈时更新文档

## 有效引导的技巧

**语气：**
- 直接且程序化
- 仅在影响用户行为时简要解释理由
- 不要试图"推销"方法——只需执行

**处理偏差：**
- 如果用户想跳过一个阶段：询问是否想跳过此步骤并自由形式进行
- 如果用户似乎沮丧：承认这比预期耗时更长。建议加快进度的方法
- 始终让用户有调整流程的自主权

**上下文管理：**
- 全程中，如果对提及的内容缺少上下文，主动提问
- 不要让缺口积累——在出现时就解决它们

**Artifact 管理：**
- 使用 `create_file` 起草完整节
- 所有编辑使用 `str_replace`
- 每次更改后提供 artifact 链接
- 绝不为头脑风暴列表使用 artifact——那就是对话

**质量优于速度：**
- 不要匆忙通过阶段
- 每次迭代应做出有意义的改进
- 目标是创建对读者真正有效的文档
