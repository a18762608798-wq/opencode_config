# 知识整理 Skill

## 概述

从当前对话中识别和提取新知识点，整理成结构化的中文 Markdown 笔记，归档到对应 Obsidian 知识库的 `00_inbox/` 目录中。

## 触发时机

1. **手动触发**：用户明确说"整理新知识"、"整理知识点"、"总结知识点"、"归档知识"、"整理笔记"等类似表述。
2. **自动触发**：对话中出现 `git commit` 或 `git push` 命令时（此时通常意味着一个工作阶段结束，顺便整理本轮对话中的新知识）。

## 知识库路径映射

| 分类 | 子类 | 目标路径 |
|------|------|----------|
| 编程 | Python | `~/sync/computer/code/program/python_note/00_inbox/` |
| 编程 | Julia | `~/sync/computer/code/program/julia_note/00_inbox/` |
| 理论 | 量子计算 | `~/sync/theory/physics/contemporary_physics/quant_comp_note/00_inbox/` |
| 理论 | 量子力学 | `~/sync/theory/physics/mechanics/Quantum mechanics/00_inbox/` |
| 理论 | 数学 | `~/sync/theory/math_note/00_inbox/` |
| 语言 | 英语 | `~/sync/tool/word_learn/00_inbox/` |

如果目标路径的 `00_inbox/` 目录不存在，先创建它。

## 工作流程

### 1. 扫描对话

回顾整个对话历史，识别其中出现的**新知识点**。知识点是指：
- 新学习的编程概念、语法特性、库的用法
- 解决某个问题的方法或技巧
- 代码实现中的关键设计决策与原因
- 量子计算/量子力学中的概念、公式、算法
- 对话中讨论的理论推导或原理解释
- 调试过程中发现的坑和解决办法

以下内容**不**视为知识点，无需整理：
- 已经熟悉的、反复使用的基础内容
- 仅作为上下文引用的已有知识
- 临时性的、一次性操作（如"帮我运行这条命令"）

### 2. 分类

将每个知识点归入最合适的类别。如果一个知识点同时涉及多个类别（比如用 Julia 实现量子算法），将编码部分归入 Julia 笔记库，理论部分归入量子计算笔记库。

### 3. 生成笔记文件

为每个知识点创建一个独立的 Markdown 文件。

**文件命名规则**：`YYYYMMDD_HHMMSS_简短描述.md`
- 使用当前时间戳
- 简短描述用中文，不超过 20 个字
- 只使用字母、数字、汉字、下划线和连字符

**文件内容模板**：

```markdown
# [知识点标题]

**日期**: YYYY-MM-DD HH:MM
**分类**: [编程/Python | 编程/Julia | 理论/量子计算 | 理论/量子力学 | 理论/数学 | 语言/英语]
**标签**: #tag1 #tag2

## 背景

[对话中涉及该知识点的上下文，一两句话说明为什么会出现这个知识点]

## 内容

[知识点的核心内容，用中文详细阐述。如果是代码，附上关键代码片段并加说明]

## 要点

- 关键点 1
- 关键点 2
- ...
```

### 4. 汇总报告

整理完成后，向用户简要报告：
- 总共识别了多少个知识点
- 每个知识点的标题和归档位置
- 如果某个知识点无法归类（不匹配任何已有知识库），询问用户是否需要单独处理

## 注意事项

- 笔记语言为中文，但代码、变量名、函数名、库名等保留英文原文。
- 一个文件只包含一个知识点，不要将多个不相关的知识点合并到一个文件。
- 如果整个对话中没有明显的新知识点，如实告知用户"本轮对话未发现需要整理的新知识点"。
- 判断一个内容是否为"新知识"时，参考的标准是：用户是否在对话中首次接触或讨论这个话题。如果不确定，倾向于整理（宁可多整理，不要遗漏）。
- 对于代码类知识点，优先保存**实际可运行的代码片段**，而非伪代码。
- 量子计算和量子力学可能有交叉（如量子态、测量等），根据对话的具体语境判断归属。如果对话侧重计算/算法层面，归入量子计算；如果侧重物理原理/公式推导，归入量子力学。
